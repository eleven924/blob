# Go vs Java IO阻塞全场景过程总结
先统一基础模型：
- Java：1:1 模型，`Thread` = OS内核线程，无用户态调度器，全靠OS调度线程
- Go：GMP模型，G(协程)、M(OS线程)、P(逻辑处理器，绑定CPU核心执行权)
分三大IO场景：**BIO阻塞IO(文件/普通Socket)、网络非阻塞IO、异步文件IO(io_uring/AIO)**

## 一、场景1：同步阻塞IO（Java BIO / Go os.File 文件读写）
### 1. Java BIO（文件、普通Socket）执行流程
1. Java Thread 调用 read/write 阻塞系统调用
2. OS将该内核线程标记为WAITING，从CPU切下，**线程卡死无法复用**
3. JVM无内置调度，这条线程彻底停滞，不能处理任何其他任务
4. 若应用内还有其他RUNNABLE线程：OS调度其他线程跑，CPU不空闲
5. 若应用所有线程全部阻塞：进程无就绪任务，分配给进程的CPU算力闲置
6. 并发大量阻塞IO：必须预开大量线程池，OS线程数量持续上涨，内核切换开销大

核心特点：线程资源、CPU执行权全部绑定，阻塞后两者同时浪费。

### 2. Go 普通文件阻塞IO（os.Read/Write）执行流程
1. G执行文件阻塞syscall，触发`entersyscall`
2. 当前M绑定的OS线程进入内核阻塞，**M被卡死，无法复用**
3. Runtime 立刻把P和阻塞M解绑，释放CPU执行权
4. 调度器优先从空闲M池取闲置OS线程绑定P；无空闲M则**新建OS线程**
5. P拿着CPU核心，继续调度其他就绪G运行（内存计算、其他逻辑），CPU不会闲置
6. 文件IO完成后，阻塞的M解除休眠，重新寻找P恢复原协程执行
7. 大量并发文件IO：OS线程数量持续上涨，但CPU算力全程充分利用

核心特点：OS线程被占用浪费，但CPU核心(P)完全解放，可跑其他协程。

## 二、场景2：网络非阻塞IO（Java NIO Netty / Go net包TCP/HTTP）
### 1. Java NIO（Selector+Channel）流程
1. Socket设置非阻塞，无数据时read直接返回，不阻塞线程
2. 单条工作线程循环调用Selector轮询所有连接事件（可读/可写）
3. 仅处理就绪连接，一条线程可管理上万连接，OS线程极少
4. 无事件时线程循环轮询，轻微消耗少量CPU，不会卡死休眠
5. 文件FileChannel不适用此机制，读文件依然阻塞线程

核心特点：靠单/少量线程轮询复用连接，规避大量阻塞线程。

### 2. Go 网络IO（netpoller epoll/kqueue）流程
1. G发起Socket读写，设置非阻塞，FD注册到runtime全局poller
2. G标记为等待状态，**当前M直接解绑P，归还到空闲M线程池**
3. P立刻切换其他G执行，原OS线程完全释放、可复用
4. 网卡数据到达，poller唤醒G，重新取M+P执行协程
5. 全程不会卡死任何OS线程，高并发下线程池数量稳定，极少新建线程

核心特点：等待期间同时释放CPU执行权+OS线程，资源利用率最高。

## 三、场景3：真正异步文件IO（Java AIO / Go io_uring）
### 1. Java AIO AsynchronousFileChannel
1. 提交读写任务，方法立即返回，线程不会阻塞
2. 内核后台完成IO，通过回调/Future通知线程处理结果
3. IO等待阶段完全释放OS线程，线程可处理其他业务
4. Linux下底层依赖libaio，生态差，生产极少使用

### 2. Go io_uring 异步文件读写
1. IO任务提交内核队列，调用立刻返回，不触发阻塞syscall
2. M解绑P放回线程池，P持续调度其他协程
3. 内核完成文件IO后通知runtime唤醒G
4. 文件IO也实现和网络IO一致的线程复用，不会暴涨OS线程

## 四、横向核心差异对照表
| IO类型                | Java BIO                          | Go 普通文件IO                     | Go 网络IO(netpoller)              |
|-----------------------|-----------------------------------|----------------------------------|-----------------------------------|
| 阻塞后OS线程状态      | 卡死休眠，不可复用                | 卡死休眠，不可复用                | 释放回线程池，可复用              |
| CPU核心(P)是否释放    | 不释放，无就绪线程则CPU闲置       | 立即释放，可跑其他协程            | 立即释放，可跑其他协程            |
| 并发阻塞时线程数量    | 随并发同步暴涨                    | 随并发同步暴涨                    | 稳定少量，几乎不新增线程           |
| 同一进程内CPU利用率   | 全阻塞则算力浪费                  | 全程高利用率                      | 全程高利用率                      |

## 五、一句话极简总结
1. 磁盘文件同步读写：Java整条线程卡死、CPU易闲置；Go仅卡死OS线程，CPU持续干活，但两者都会创建大量OS线程。
2. 网络请求：Java NIO靠轮询少线程复用；Go内置poller，等待时直接归还OS线程，资源最优。
3. 异步文件io_uring/AIO：两者都能实现线程完全释放，解决高并发文件线程爆炸问题。
